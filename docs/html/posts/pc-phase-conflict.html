同一个 PC 的相位冲突

跨编译单元的函数调用如何污染训练标签 —— 同一个 add (%rdi),%ecx，三种 stride（+4, +16, +64）；同一个 add (%rdi,%rdx,4),%eax，顺序访问和随机访问交替出现。 不依赖任何 noinline hack，只用多文件项目的日常编译行为演示相位冲突的产生与消解。

1

跨 .o 调用 = 天然不可内联

无 LTO 时，编译器只看到调用方的 .c 文件，函数体在另一个 .o 里。所有跨 TU 调用必然共享同一个 PC。

2

同一 PC ≠ 同一模式

stride 参数由调用方传入，indices 数组内容决定访问模式。同一个 load 指令，行为完全取决于调用上下文。

3

内联是天然解药

函数体在同 .c 文件时，gcc -O2 内联到各 call site→不同 PC→标签不冲突。跨文件时需 LTO 或引入 call-chain 上下文。

实验设置：跨编译单元模拟真实代码库

真实项目（SPEC、Linux kernel、任何多文件 C 项目）中，函数分布在不同的 .c 文件。编译时各自生成 .o，链接时拼在一起。跨 .o 的函数调用在 -O2 下无法内联（除非开启 LTO）。

worker.h — 函数声明
worker.c — 实现在另一个编译单元
main.c — 调用方
$ gcc -no-pie -O2 -c worker.c -o worker.o
$ gcc -no-pie -O2 -c main.c -o main.o
$ gcc -no-pie worker.o main.o -o demo

-no-pie 让代码段 PC 固定，方便反汇编时精确定位。与 SPEC 的编译方式一致。

案例 1：同一 PC，三种不同的 stride

PC 0x4013ea 跨 TU sum_strided — 被三处调用，stride 各不相同

worker.c（另一个编译单元）

// 函数体在 worker.o 中，main.o 编译时完全看不到 int sum_strided(int *arr, int n, int stride) { int sum = 0; for (int i = 0; i < n; i += stride) { sum += arr[i]; // ← 关键的 Load PC } return sum; }

main.c 反汇编（调用方）

; 三处 call 全部指向同一个 PC — 因为函数体在另一个 .o 里 401166: call 4013d0 <sum_strided> ; stride=1 → 步进 +4/iter 401193: call 4013d0 <sum_strided> ; stride=4 → 步进 +16/iter 4011c0: call 4013d0 <sum_strided> ; stride=16 → 步进 +64/iter

worker.o 中 sum_strided 的反汇编

; sum_strided 内部的循环 —— stride 由 r8 寄存器控制 4013d8: movslq %edx,%r8 ; stride → r8 4013df: shl $0x2,%r8 ; r8 = stride * 4 (字节偏移) 4013e8: add %edx,%eax ; i += stride 4013ea: add (%rdi),%ecx ; ← 唯一的 Load PC: 0x4013ea 4013ec: add %r8,%rdi ; rdi += stride*4 4013f1: jg 4013e8 ; loop
关键机制
循环变量 i 由 rdi 指针的移动来隐式跟踪——每次迭代 rdi += stride*4。三处调用传入不同的 stride → r8 分别为 4, 16, 64 → 同一 PC 0x4013ea 看到的地址 delta 分别是 +4, +16, +64。

PC 0x4013ea 的 profiler 记录中混合了三种 stride。如果 next_line 对顺序访问最优，它对稀疏访问可能完全无用——选出的 best_prefetch 是对三种模式的折中。

案例 2：同一 PC，顺序访问 vs 随机访问

这是更极端的情况：同一个 load PC 在顺序遍历和随机跳转之间切换。同样跨编译单元，无 hack。

PC 0x40142f 顺序⇄随机 sum_indirect — indices 内容决定访问模式

worker.c

// 通过 indices 间接访问 arr — indices 的内容决定 arr 的访问模式 int sum_indirect(int *arr, int *indices, int n) { int sum = 0; for (int i = 0; i < n; i++) { sum += arr[indices[i]]; // ← 关键的 Load PC } return sum; }

main.c 调用

// Phase A: indices = [0,1,2,...] → 顺序访问 arr for (int i = 0; i < 512; i++) indices[i] = i; volatile int r4 = sum_indirect(data, indices, 512); // Phase B: indices 随机打乱 → 随机访问 arr shuffle(indices); volatile int r5 = sum_indirect(data, indices, 512);

sum_indirect 反汇编

401418: movslq %edx,%rdx 40141d: lea (%rsi,%rdx,4),%rcx ; &indices[n] (循环边界) 401428: movslq (%rsi),%rdx ; rdx = indices[i] 40142b: add $0x4,%rsi ; i++ (遍历 indices) 40142f: add (%rdi,%rdx,4),%eax ; ← 唯一的 Load PC: 0x40142f 401432: cmp %rsi,%rcx ; arr[indices[i]] 401435: jne 401428 ; loop

适合 next_line：严格的 +4 stride

indices[2]=501 arr[501] @ ...f74 ← 又大跳

为什么会这样？
arr[indices[i]] 的地址取决于 indices[i] 的值——这是运行时数据，编译器完全无法预知。

同一个 PC 0x40142f 在 profiler 中会收到两段截然不同的 AMAT 记录：一段是完美顺序（hit 率高），一段是随机跳转（miss 率高）。单一 prefetcher 标签无法同时描述两者。

这种模式在真实代码中极其常见：稀疏矩阵的列索引、哈希表的 bucket 链、图的邻接表遍历——只要是"通过索引数组间接访问另一个数组"，就存在这个潜在问题。

编译器行为：何时消解，何时不消解

自然消解：同编译单元内被内联

如果 sum_strided 定义在 main.c 中（同 TU），gcc -O2 会将其内联。三个 call site 各自生成独立的 add 指令——三个不同的 PC。profiler 标签不会冲突。

无法消解的真实场景

场景为什么编译器无法拆分常见程度
跨编译单元调用 无 LTO 时编译器看不见函数体，只能生成 call 指令 几乎所有多文件项目
函数指针 / 虚函数 编译时无法确定调用目标；间接调用 (call *%rax) 必须走原 PC qsort 回调、C++ 虚函数、插件系统
动态库函数 .so 在运行时通过 PLT 链接，编译时只有 stub libc、第三方 .so
递归函数 无法完全内联（无限展开不可能） 树遍历、解析器、图算法
过大函数 超过 GCC 内联代价阈值，即使同 TU 也不内联 SPEC 巨型解释器函数（如 S_regmatch）

SPEC CPU2006 正是跨 TU 场景——400.perlbench 由数百个 .c 文件编译成 .o 再链接。跨 TU 调用无处不在。函数指针也同样存在（Perl 的 opcode dispatch）。

对 Profiling Pipeline 的影响

当前状态

Stage 6 对每个 PC 产出单一 best_prefetch 标签。在跨 TU 场景下，PC 的 AMAT 记录混合了多种模式。选出的"最优"实际上是对不同步长/模式的折中值。

排查方法

当某个 benchmark 的 GBDT 精度异常低时：

1. 检查 AMAT 分布 — 该 PC 在不同 prefetcher 下的 AMAT 是否都没有明显最优？如果有多个 prefetcher 打平，说明该 PC 的模式不稳定。
2. 检查调用来源 — objdump -d binary | grep 'call.*<func>' 看该函数被多少处调用。如果多于 1 处，且参数不同，就可能存在相位混合。
3. 检查间接访问模式 — 该 PC 是否通过 indirection 数组（arr[indices[i]]）访问数据？indices 的内容在不同阶段是否变化？

可能的改进方向

方案适用场景复杂度
PC + call-chain 上下文 同一函数被多处不同参数调用 需修改 profiler 记录 call stack
按时间分片标签 同一函数在不同执行阶段模式不同 需 SimPoints 级标签划分
多标签 / soft label PC 存在内在多义性 训练目标从分类改为概率分布
开启 LTO 编译 消除跨 TU 内联障碍 编译流程变更，SPEC 需适配

可复现实验

三文件完整源码（worker.h + worker.c + main.c）：

# worker.h
int sum_strided(int *arr, int n, int stride);
int sum_indirect(int *arr, int *indices, int n);

# worker.c — 另一个编译单元
int sum_strided(int *arr, int n, int stride) {
  int sum = 0;
  for (int i = 0; i < n; i += stride) sum += arr[i];
  return sum;
}
int sum_indirect(int *arr, int *indices, int n) {
  int sum = 0;
  for (int i = 0; i < n; i++) sum += arr[indices[i]];
  return sum;
}

# main.c — 调用方：3 种 stride + 2 种 indices 模式
# (完整代码见 snippets/)
# 编译 & 验证
$ gcc -no-pie -O2 -c worker.c -o worker.o
$ gcc -no-pie -O2 -c main.c -o main.o
$ gcc -no-pie worker.o main.o -o demo
$ objdump -d demo | grep 'call.*sum_strided'
# 输出三行 call 4013d0 <sum_strided> — 全部指向同一地址
$ objdump -d demo | grep 'call.*sum_indirect'
# 输出两行 call 401410 <sum_indirect> — 全部指向同一地址

-no-pie 下代码段 PC 固定。stack 地址因 ASLR 每次不同，但 delta 值和反汇编完全可复现。

本文源于对 "per-PC label 在真实多文件代码库中是否可靠" 的追问。结论：跨编译单元的函数调用是最常见且最自然的相位冲突来源——编译器无 LTO 时完全无法内联，同一个 PC 的 profiler 标签必然混合多种访存模式。了解这一点有助于在排查训练精度异常时定位根因。
