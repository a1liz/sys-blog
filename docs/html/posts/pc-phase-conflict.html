同一个 PC 的相位冲突

完整分类与可复现案例 —— 同一 load PC 在不同执行阶段呈现出截然不同的访存模式。 两大类 9 种情况：编译器因素（跨文件编译、函数指针等 5 种）和语义因素（参数变化、间接索引等 4 种）。

先解释几个缩写

TU（Translation Unit / 编译单元）—— 一个 .c 文件经过预处理器展开后的产物。gcc -c foo.c 编译一个 TU，产出 foo.o。大多数项目由几十上百个 TU 组成，编译时每个 TU 各自独立处理。
LTO（Link-Time Optimization / 链接时优化）—— 把优化推迟到链接阶段。不加 LTO 时，gcc -c a.c 完全看不到 b.c 里的函数体。加 -flto 后，链接器拿到所有 .o 的中间表示，可以跨文件内联。
PGO（Profile-Guided Optimization / 剖面引导优化）—— 先跑一遍程序收集热点数据，再基于这份数据重编译。可以帮编译器判断"这个函数指针实际指向谁"或"这个函数值不值得内联"。
PLT（Procedure Linkage Table / 过程链接表）—— 动态库（.so）的跳板。程序调用 memcpy 时先跳到 PLT 里的 stub，stub 再跳到真正的实现。
A

编译器因素

多个不同的调用上下文被迫共享同一个 PC。编译器做不到或不愿为它们生成独立副本。

B

语义因素

函数自身逻辑就包含多种访存模式。不是调用问题——一个 call site 内部的 PC 天然多义。

?

能否消解

A 类大部分可通过 LTO/PGO 消解。B 类基本不能，只能在训练数据层面用 soft label 或时间分片应对。

类别 A：编译器因素 — 多个调用上下文被迫共享一个 PC

这类冲突的本质：程序员写了多个不同的调用，每个调用本应有各自的访存模式，但编译器做不到或选择不为它们生成独立的副本，多个调用被迫走到同一个 PC。

A1：跨编译单元调用（无 LTO）可消解：LTO

本质：函数体在另一个 .c 文件里。不加 LTO 时，编译器在处理调用方这个 TU 的时候完全看不见被调函数的函数体，只能生成一个 call 指令指向外部符号。所有调用方共享同一个 PC。

PC 0x4013ea A1·跨 TU sum_strided — worker.c 和 main.c 分文件编译
worker.h — 函数声明
worker.c — 实现在另一个编译单元
main.c — 调用方

worker.c（另一个编译单元）

// 函数体在 worker.o 中，main.o 编译时完全看不到 int sum_strided(int *arr, int n, int stride) { int sum = 0; for (int i = 0; i < n; i += stride) { sum += arr[i]; // ← 关键的 Load PC } return sum; }

main.o 反汇编（调用方）

; 三处 call 全部指向同一地址 — 函数体在另一个 .o 里 401166: call 4013d0 <sum_strided> ; stride=1 → 步进 +4/iter 401193: call 4013d0 <sum_strided> ; stride=4 → 步进 +16/iter 4011c0: call 4013d0 <sum_strided> ; stride=16 → 步进 +64/iter

sum_strided 内部反汇编

4013d8: movslq %edx,%r8 ; stride → r8 4013df: shl $0x2,%r8 ; r8 = stride * 4（字节偏移） 4013e8: add %edx,%eax ; i += stride 4013ea: add (%rdi),%ecx ; ← 唯一的 Load PC：0x4013ea 4013ec: add %r8,%rdi ; rdi += stride*4 4013f1: jg 4013e8 ; loop
机制：循环变量 i 由 rdi 指针的移动来隐式跟踪——每次迭代 rdi += stride*4。三处调用传入不同的 stride → r8 分别为 4、16、64 → 同一 PC 0x4013ea 看到的地址 delta 是 +4、+16、+64。

消解：加 -flto 重新编译，链接器可以在链接时把 sum_strided 内联到三处调用，各自生成独立 PC。

A2：函数指针 / 虚函数部分可消解：PGO

本质：call *%rax 跳到哪去，编译器静态分析推不出来。即使函数体在同一个 TU 内、编译器完全看得到，也不知道该内联谁。

PC 0x401324 A2·函数指针 cmp_func — qsort 通过指针回调

代码

// cmp_func 的地址被传给 qsort。qsort 内部通过函数指针调用它。 int cmp_func(const void *a, const void *b) { int va = *(const int *)a; // ← Load PC #1 int vb = *(const int *)b; // ← Load PC #2（cmp 指令隐含） return (va > vb) - (va < vb); } // ... qsort(data, 512, sizeof(int), cmp_func);

cmp_func 反汇编

0000000000401270 <cmp_func>: 401274: mov (%rsi),%eax ; ← Load PC #1：b 的值 401276: cmp %eax,(%rdi) ; ← Load PC #2：a 的值（cmp 隐含 load） 401278: setg %al 40127b: setl %dl 401284: sub %edx,%eax 401286: ret

main 中的调用

; cmp_func 的地址被加载到 rcx，传给 qsort 401133: mov %rbx,%rdi 401136: mov $0x200,%esi 40113b: call 401060 <qsort@plt> ; qsort 内部通过 *rcx 回调 cmp_func
机制：PC 0x401274 和 0x401276 在 qsort 执行期间被调用几万次。a 和 b 的地址由 qsort 内部的快排分区算法决定——一会儿比较 pivot（数组中央），一会儿分区交换（数组两端），一会儿递归子数组。同一个 load PC 的地址序列完全随机。

消解：PGO 可以先跑一遍程序，收集"这个指针 99% 指向 cmp_func"的热点数据，下次编译时生成 if (fp == cmp_func) call cmp_func else call *fp（speculative devirtualization）。但这个内联是概率性的，不保证一定被编译器采纳。

A3–A5：其余编译器因素

#场景本质能否消解
A3 动态库函数 .so 运行时加载，编译时只有一个 PLT stub。所有 call memcpy@plt 指向同一个 stub PC。 可消解 静态链接（.a 替代 .so）
A4 函数体过大 即使同 TU、编译器看得到，几千行的函数内联代价远超收益，gcc 拒绝内联。如 perlbench 的 S_regmatch。 部分 LTO + PGO 裁掉冷路径后内联热路径
A5 递归函数 理论上不可能展开为有限层。同一 PC 在不同递归深度访问不同的树节点。 部分 尾递归可优化为迭代，一般递归不可

类别 B：语义因素 — 函数自身逻辑天然多模

这类冲突的本质：不是调用方的问题。即使只有一个 call site，编译器做了它能做的一切，函数内部的 PC 仍然会在不同时间做不同的事——因为这是函数本身的逻辑决定的。

B1：同一 call site，参数随时间变化不可消解

本质：只有一个call site（汇编里一条 call 指令），没有多个调用方。但参数值在不同时间传入的值不同，导致同一 PC 的访存模式随时间漂移。

PC 0x4013ea B1·参数变化 同一个 call，不同时间传不同 stride
// 只有一个 call site，但 stride 来自变化的数据源 int strides[] = {1, 4, 16}; for (int phase = 0; phase < 3; phase++) { sum_strided(data, 512, strides[phase]); // 同一个 call }
为什么编译器不能消解：为 stride=1、stride=4、stride=16 各生成一份 sum_strided 的副本，要求编译器在编译时就知道 strides[] 的全部可能取值——这不现实。更常见的真实场景：图像处理的 kernel_size 由用户输入指定，同一个高斯模糊函数在 kernel=3 和 kernel=15 时访存跨度完全不同。

B2：间接索引访问（data-dependent index）不可消解

本质：访问地址 = base + indices[i] * sizeof(T)。indices 的内容是运行时数据，编译器完全无法预测。

PC 0x40142f B2·间接索引 顺序 ⇄ 随机 sum_indirect — indices 内容决定访问模式

代码 & 反汇编

// 通过 indices 间接访问 arr — indices 内容决定 arr 的访问模式 int sum_indirect(int *arr, int *indices, int n) { int sum = 0; for (int i = 0; i < n; i++) { sum += arr[indices[i]]; // ← 关键的 Load PC } return sum; }
401428: movslq (%rsi),%rdx ; rdx = indices[i]（符号扩展） 40142b: add $0x4,%rsi ; i++（遍历 indices 数组本身是顺序的） 40142f: add (%rdi,%rdx,4),%eax ; ← 唯一的 Load PC：arr[indices[i]] 401432: cmp %rsi,%rcx 401435: jne 401428 ; loop

为什么编译器不能消解：indices[i] 的值在运行时才确定。编译器没有办法知道 Phase A 和 Phase B 的区别——它只能看到一条 add (%rdi,%rdx,4),%eax，rdx 是循环中从 (%rsi) 加载的，值完全由运行时数据决定。

这种模式在真实代码中极其常见：稀疏矩阵的列索引遍历、哈希表的 bucket chain、图的邻接表——只要是"通过索引数组间接访问另一个数组"，就属于此类。

B3–B4：其余语义因素

#场景本质能否消解
B3 多阶段算法 同一循环在不同阶段处理不同性质的数据。LZ 解压：literal copy（顺序）→ back-reference（按 offset 随机跳转）。 不可 只能时间分片标签
B4 交替数据结构 同一循环交替访问 A[i] 和 B[i]，模式在两者间快速切换。 部分 loop fission 可拆分，但破坏局部性

两类对比

类别 A（编译器因素）类别 B（语义因素）
问题来源 多个调用方被迫共享同一个 PC 函数自身逻辑就包含多种模式
调用方数量 多个（或通过指针间接调用） 可以只有一个
能否被编译优化消解 部分可以（LTO、PGO、静态链接） 大部分不能
应对策略 LTO、PGO、合并源文件 soft label、时间分片、call-chain 特征

对 Profiling Pipeline 的影响

当前 Stage 6 对每个 PC 产出单一 best_prefetch 标签。在上述 9 种情况下，这个标签都是多种模式的折中值。

排查步骤

1. 检查 AMAT 分布 — 该 PC 在各 prefetcher 下的 AMAT 是否都没有明显最优？多个 prefetcher 打平说明模式不稳定。
2. 检查调用来源 — objdump -d binary | grep 'call.*<func>' 看该函数是否被多处调用。多于 1 处 → 可能是 A1。
3. 检查间接访问 — 该 PC 是否通过 arr[indices[i]] 模式访问？indices 在不同阶段是否变化？→ 可能是 B2。
4. 分情况应对 — A1 尝试 LTO 重编译。A2 尝试 PGO。B 类考虑 soft label 或引入 call-chain 上下文作为训练特征。

完整源码在 snippets/ 下：

# A1: 跨编译单元 — 3 种 stride 共享 1 个 PC
$ gcc -no-pie -O2 -c worker.c -o worker.o
$ gcc -no-pie -O2 -c main.c -o main.o
$ gcc -no-pie worker.o main.o -o demo
$ objdump -d demo | grep 'call.*sum_strided'
# 输出三行 call 4013d0 <sum_strided> — 全部指向同一地址

# A2: 函数指针 — qsort 回调 cmp_func
$ gcc -no-pie -O2 -g qsort_demo.c -o qsort_demo
$ objdump -d qsort_demo | grep -A8 '<cmp_func>'
# 两条 load PC 在 qsort 内部被通过指针重复调用几万次

-no-pie 下代码段 PC 固定，可复现。stack 地址因 ASLR 每次不同，但 delta 值一致。

本文源于对 "per-PC label 在真实代码库中是否可靠" 的系统性追问。PC 相位冲突可归为两大类 9 种情况：编译器因素（5 种，部分可消解）和语义因素（4 种，基本不可消解）。了解每种情况的机制和边界，是排查训练精度异常的基础。
