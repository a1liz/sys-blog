AMAT 如何揭开 CPU 预取的秘密

从 ChampSim 逐指令级 Profiling 到 Prefetcher 选择 —— 通过 AMAT 在 14 种预取配置中为 4,202 个 Load PC 选出最优策略。 附带 400.perlbench 真实汇编级案例，解释为什么 next_line 能赢 46%，而 ip_stride 在某些指针链上比什么都不做还差。

1

每个 Load PC 的访存模式不同

顺序扫描数组、指针链跳转、固定步长遍历结构体 —— 单一 prefetcher 无法适应所有模式。

2

AMAT 是最小控制变量实验

同一 trace × 不同 prefetcher → AMAT 差异纯粹来自 hit/miss 比例，AMAT 最低者为最优。

3

静态度量 → 动态行为 可学习

编译后的机器码在同一输入下总是走同样的路径。汇编上下文 → 最优 prefetcher 的映射是 ML 可学的。

AMAT 是什么

AMAT（Average Memory Access Time） 是一条 load 指令每次访问内存的平均延迟。

AMAT = (hit_count × HIT_LATENCY + miss_count × FILL_LATENCY) / access_count
结果延迟
Cache HitHIT_LATENCY（几个 cycle）
Cache MissFILL_LATENCY（几十到上百 cycle）

Prefetcher 的价值就是把 Miss 变成 Hit。AMAT 越低 → hit 越多 → 程序跑得越快。

ChampSim 是一款 cycle-level 缓存模拟器。编译时加 -DHINT_PROFILING，在 CACHE::try_hit() 路径上嵌入 profiler：

// src/cache.cc — 每一次 load 经过这里
bool CACHE::try_hit(const tag_lookup_type& handle_pkt) {
  // ... tag 查找，判定 hit/miss ...

  uint64_t latency = hit ? HIT_LATENCY.count() : FILL_LATENCY.count();
  PROFILER_RECORD_ACCESS(handle_pkt.ip, /*repl*/0, /*pref*/0, hit, latency);

  // ... 然后才触发 prefetcher，做预取 ...
}

程序跑完 100M 条指令后，profiler flush 每个 PC 的累计统计：

// src/profiler.cc — flush 时算 AMAT
double avg_amat = rec.total_latency / rec.access_count;  // 就是 AMAT

案例 1：next_line 大胜 46% — 结构体字段的连续访问

Perl_pp_match PC 0x463148 AMAT Gap +46%
0x46313c: mov %rax,0x78(%rsp)
0x463141: xor %eax,%eax
0x463143: movzbl 0x24(%r8),%eax ← 结构体字段 +0x24
>>> mov 0x50(%r8),%r14 ← 结构体字段 +0x50 (LOAD)
0x46314c: mov %eax,%edx
0x46314e: and $0x3,%edx

r8 指向匹配引擎状态结构体。两个字段偏移递增 +0x2C——同一结构体内部的相邻成员。

next_line:1 398.4 ← 最优
no:1 580.4 +46%
ip_stride:2 600.3
stride:1 673.4
为什么？ 第一次 miss 后 next_line 把紧接着的下一行预取进来，后续 +0x50 直接命中。而 no 每次 miss 都等完整 fill latency。差距高达 182 cycle。

案例 2：next_line 再胜 41% — 正则引擎的循环状态检查

S_regmatch PC 0x4898cb AMAT Gap +41%
0x4898bf: mov %r14,%rcx
0x4898c2: mov %eax,0x18(%rsp)
0x4898c6: jmp 488378 <S_regmatch+0x118>
>>> cmp 0x9753e(%rip),%r14 ← 对比 PL_regeol 边界 (LOAD)
0x4898d2: jb 488395 <S_regmatch+0x135>
0x4898d8: jmp 488375 <S_regmatch+0x115>

Perl 正则引擎核心匹配循环。每次迭代顺序访问全局状态变量 PL_regeol。

next_line:1 375.0 ← 最优
ampm:1 527.0 +41%
va_ampm_lite:1 527.0
为什么？ 正则引擎主循环是高度可预测的顺序流。next_line 的最简策略完美匹配。复杂的 ampm 和 va_ampm_lite 反而因为过度预取或错误预测浪费了带宽。

案例 3：ip_stride 胜出 — 方法分发的指针链跳转

Perl_pp_method_named PC 0x4689ec AMAT Gap +14%
0x4689e1: mov 0xb7db0(%rip),%rax ← PL_op 全局表
0x4689e8: mov 0x28(%rax),%rbp ← op→op_next (+0x28)
>>> mov 0x0(%rbp),%rax ← op_next→op_type (+0x00, LOAD)
0x4689f0: mov 0x18(%rax),%rax ← op_type→op_ppaddr (+0x18)
0x4689f4: mov %eax,0x4(%rsp)

Perl 的方法分发链：偏移量不断变化——+0x28 → +0x00 → +0x18——不是顺序的。

ip_stride:2 411.7 ← 最优
no:1 468.0 +14%
next_line:1 486.8 比 no 还差
为什么？ ip_stride 在 PC 级别学习了固定步长模式——这个 load 高频重复执行。而 next_line 比 no 还差：预取了错误的行，占用了缓存带宽，拖累了真正需要的数据。

案例 4：什么都不做反而最好

Perl_sv_catsv PC 0x4a1288 AMAT Gap +17%
>>> mov 0x8(%rbx),%rdx ← 随机指针解引用 (LOAD)
0x4a128c: test %rdx,%rdx
0x4a128f: je 4a12a0
no:1 375.0 ← 最优
next_line:1 437.5 +17%
ip_stride:1 450.0
为什么？ 当访存模式完全随机时，任何预取都是浪费——预测准确率接近 0%，反而占用缓存空间和带宽。有时候什么都不做就是最好的策略。

如何变成训练数据

Stage 7 将汇编上下文（Stage 5）与最优标签（Stage 6）合并成 instruction-tuning 格式：

"instruction": "Given the following x86-64 assembly context for a load instruction in function 'Perl_pp_match', predict the optimal prefetch policy. Instructions before load: 0x46313c: mov %rax,0x78(%rsp) 0x463141: xor %eax,%eax 0x463143: movzbl 0x24(%r8),%eax Load instruction: mov 0x50(%r8),%r14 Instructions after load: 0x46314c: mov %eax,%edx 0x46314e: and $0x3,%edx ...", "label": "next_line:1", "pc": "0x463148"

模型学到：看到 r8+0x24 然后 r8+0x50 这种相邻偏移 → 选 next_line。看到 rip→+0x28→+0x00→+0x18 的指针链 → 选 ip_stride。

核心洞察

AMAT 是控制变量实验同一 trace × 不同 prefetcher → AMAT 差异纯粹来自 hit/miss 比例差异
Prefetcher 是近似函数f(PC_history, global_pattern) → address — 每个 prefetcher 的近似能力不同
最低 AMAT = 最优近似对特定 PC 的访存模式，AMAT 最低的 prefetcher 提供了最好的近似
汇编上下文稳定编译后的机器码在同一输入下走同样路径 → 静态度量到动态行为的映射是可学习的
